# Cloud Ops Agent Guidelines

## 目标

本目录是个人云服务器的通用运维与服务归档入口。所有通用服务、跨项目服务、部署规范和方案记录优先归档到这里，不绑定到某个具体项目仓库。

当前服务器主要用途：个人联调、通用工具服务、N2 以及后续其他项目的共享后端能力。

## 执行原则

- 云端操作前先阅读 `/home/ubuntu/cloud-ops/AGENTS.md` 和相关 `docs/*.md`。
- 通用服务默认独立部署，不把服务能力耦合到 N2 或其他单一项目。
- 服务代码、配置、运行数据分离：`/opt/<service>`、`/etc/<service>.env`、`/var/lib/<service>`。
- 密码、API Key、Token、私钥内容不得写入本目录文档、脚本或 Git 仓库。
- `.env` 只记录在 `/etc` 或服务私有目录，文档只记录字段名和用途。
- 执行重启、发布、清数据、改防火墙、改 systemd 前，需要明确目的、影响范围和回滚方式。
- 变更完成后，记录验收命令和结果；重要方案调整写入 `history/`。

## 标准变更流程

每次部署、改配置或调整 systemd 服务，按以下流程执行：

1. 阅读本文件和对应服务文档。
2. 查看当前服务状态、端口占用和最近日志。
3. 备份关键配置，尤其是 `/etc/<service>.env`。
4. 执行改动。
5. 重启或 reload 服务。
6. 执行 `/healthz` 和关键接口验收。
7. 查看 `journalctl`，确认没有异常和密钥泄露。
8. 将重要变更写入 `history/YYYY-MM-DD-<service>-<action>.md`。

## 服务归档要求

新增或调整通用服务时，至少补充：

- 服务目标与使用方
- 部署目录与 systemd 服务名
- 端口、健康检查、日志查看命令
- 配置项清单，不包含真实密钥
- 验收命令
- 回滚或停止方式
- 后续扩展点

## 硬性规则

- 通用服务不放进 `/home/ubuntu/n2-server/`。
- 真实密钥只放 `/etc/<service>.env` 或明确的服务私有配置文件。
- 缺关键配置时服务应启动失败，不用默认值或假服务掩盖问题。
- systemd 托管的 HTTP 服务必须提供 `/healthz`。
- 生成、上传、数据库等状态类数据不得混入服务代码目录。
